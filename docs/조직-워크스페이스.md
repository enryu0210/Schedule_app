# 조직(팀/기업) 워크스페이스

> 상태: 1차 구현 완료(웹). 실사용 검증 전.
> DB 스키마: `supabase/org-schema.sql` (Supabase SQL Editor 에서 실행 완료)

---

## 1. 왜 만들었나 (실제로 겪은 문제)

사용자가 속한 조직에서 두 가지 손실이 실제로 발생하고 있었다.

1. 일정을 몰라서 **약속·회의를 잡을 때마다 매번 물어봐야 한다.**
2. 관리자가 누가 언제 일하는지 파악이 안 되어 **쓸모없는 미팅이 여러 번 잡힌다.**

## 2. 흐름은 '고리'다 (설계의 핵심)

```
팀원이 각자 개인 일정을 적는다
  → 관리자가 그걸 모아 본다 (누가 언제 비는지)
  → 관리자가 그걸 보고 조직 시간표를 짜서 배포한다
  → 팀원이 배포된 조직 시간표를 본다
```

**순서가 중요하다. 관리자의 "열람"이 "배포"의 전제조건이다.**
배포 기능만 먼저 만들면 관리자는 여전히 깜깜이로 일정을 짜게 되고,
원래 고통(쓸모없는 미팅)이 그대로 남는다.

## 3. 프라이버시 — 경계선을 테이블에 새긴다

일정공방은 **개인용 앱**이다. 사람들이 사적인 일정을 여기 적는다.
경계를 안 그으면 사람들은 개인 일정을 앱에 안 적기 시작하고, 그러면 조직 시간표도 같이 부실해진다.
**도구가 스스로를 망친다.**

그래서:

- **개인 프리셋(`user_data`)은 조직에서 절대 보이지 않는다.** RLS 정책을 한 줄도 바꾸지 않았다.
- 조직에 보이는 것은 팀원이 **직접 "공유"를 누른 프리셋의 사본**(`org_shared_schedules`)뿐이다.
- 감추고 싶은 일정은 블록 이름을 "개인일정" 처럼 적으면 된다 —
  **공개 범위를 사용자가 이미 쥐고 있다.** (그래서 별도의 '바쁨/한가함만 표시' 모드를 만들지 않았다)

## 4. 데이터 모델

| 테이블 | 역할 |
|---|---|
| `organizations` | 조직 (이름 + 8자리 **초대 코드**) |
| `org_members` | 조직 ↔ 사용자 + 역할(`admin` / `member`) + 조직 내 표시 이름 |
| `org_shared_schedules` | 팀원이 조직에 공유한 시간표 **사본** (1인 1조직당 1개) |
| `org_plans` | 관리자가 배포한 조직 공용 시간표 (조직당 1장) |

**정규화하지 않은 이유**: 프리셋은 지금 `user_data.presets` 에 통짜 jsonb 로 있고
웹·위젯·안드로이드가 **전부 그 구조에 의존**한다. 조직 기능을 위해 그걸 뜯으면
아직 아무도 안 써본 기능 때문에 개인 앱 전체가 흔들린다.
→ **사본 방식**으로 시작하고, 실제로 쓰이는 게 확인되면 그때 정규화한다.
사본의 대가(원본을 고치면 사본이 뒤처짐)는 자동 갱신으로 메웠다(§6).

## 5. 함정 (실제로 밟을 뻔한 것)

1. **RLS 무한 재귀.** `org_members` 의 정책이 "같은 조직 사람이면 보인다"인데,
   이를 확인하려고 다시 `org_members` 를 조회하면 **정책을 평가하려고 정책을 평가하는** 고리가 생겨
   쿼리가 통째로 실패한다. → `security definer` 함수(`is_org_member`, `is_org_admin`)로 고리를 끊었다.
   (`set search_path = public` 을 반드시 박는다 — definer 함수는 권한이 세서 스키마 바꿔치기 통로가 된다)

2. **이메일 초대는 쓸 수 없다.** 카카오 KOE205 를 풀 때 Supabase 에서
   **"Allow users without an email"** 을 켰다 → **이메일이 아예 없는 사용자가 존재한다.**
   그래서 초대는 8자리 **코드**를 알려주고 입력하게 한다.

3. **"못 읽음"을 빈 화면으로 뭉개면 안 된다.** 공유된 시간표가 네트워크 문제로 안 보이는데
   관리자가 "이 사람은 안 냈네" 하고 일정을 짜면 사고다.
   → `orgStorage.ts` 는 실패 시 **예외를 던지고**, 화면은 오류를 표시한다.
   구성원 목록에도 **"공유함 / 미공유"** 를 명시해, 안 낸 사람을 한가한 사람으로 착각하지 않게 했다.

4. **조직 생성은 반드시 원자적으로.** '조직 생성'과 '나를 관리자로 등록'이 따로 놀면
   중간에 실패했을 때 **주인 없는 조직**이 남는다. → RPC 함수(`create_org`) 안에서 한 번에 처리.

## 6. 공유 사본 자동 갱신

공유되는 것은 사본이라, 팀원이 개인 계획표를 고쳐도 공유본은 옛날 그대로 남는다.
**팀원은 고쳤다고 생각하는데 관리자는 옛 시간표를 보고 회의를 잡는다** — 아무도 모른 채 사고가 난다.
사람이 '갱신' 버튼 누르는 것에 의존하면 안 된다.

→ `hooks/useSharedScheduleSync.ts`: 개인 프리셋이 바뀌면, 그 프리셋을 공유해 둔
**모든 조직의 사본을 자동으로 따라 갱신**한다. (편집 후 1.2초 디바운스, 내용이 같으면 쓰지 않음)

**일부러 안 하는 것 — 프리셋을 지웠을 때 공유본까지 지우지는 않는다.**
클라우드 로딩 도중 `presets` 가 잠깐 빈 배열인 순간이 있어서, 그걸 '삭제'로 오해하면
팀원의 공유 시간표가 통째로 날아간다. **되돌릴 수 없는 일은 자동으로 하지 않는다.**
지우는 것은 '공유 내리기'로 사용자가 직접.

## 7. 조직 시간표는 조직 페이지에서 직접 짠다

처음엔 "개인 프리셋을 만들어 조직에 배포"하는 방식이었는데,
조직 시간표를 짜려고 **내 개인 계획표를 먼저 조직용 초안으로 어지럽혀야** 해서 순서가 억지스러웠다.
→ `OrgPlanEditor` 에서 그리드로 바로 짜고, **고치는 즉시 자동 저장(=배포)** 된다.

- 자동 저장은 0.8초 디바운스 (드래그 한 번에 서버 쓰기가 수십 번 나가는 것을 막는다)
- **저장 상태를 반드시 표시한다** ("저장 중… / 배포됨 ✓ / 저장 실패").
  실패를 조용히 넘기면 관리자는 배포된 줄 알고 창을 닫는다.
- 편집 중이면 서버 값이 내려와도 덮어쓰지 않는다(`dirty` 플래그).
  관리자가 둘일 때 타이핑 도중 내 입력이 사라지는 게 최악이다.
  단, 이는 완전한 충돌 해결이 아니다 — **나중에 저장한 쪽이 이긴다.** 관리자가 보통 1명이라 지금은 이걸로 충분.
- 조직 시간표 편집을 위해 `BlockEditor` 에 **삭제 버튼(선택 prop)** 을 추가했다.
  그래프 뷰로만 편집하는데 삭제는 목록 뷰에만 있어서 **블록을 지울 방법이 아예 없었다.**


## 9. Realtime 구독 ✅ (`useOrg.tsx`)

관리자가 배포하면 팀원 화면이, 팀원이 공유하면 관리자 화면이 **새로고침 없이** 바뀐다.
(DB 발행은 `org-schema.sql` 에서 이미 등록해 뒀다 — 앱이 듣기만 하면 됐다)

현재 조직을 대상으로 채널 하나를 걸고, `org_shared_schedules` / `org_plans` 의 변경을 받는다.

- **페이로드를 믿지 않고 다시 읽는다**(`reloadCurrentOrg`). 위젯에서 검증된 방식 —
  알림 순서가 뒤바뀌거나 놓쳐도 항상 최신으로 수렴한다.
- RLS 가 그대로 적용된다 — 남의 조직 변경은 애초에 전달되지 않는다.
- **폴링 안전망은 두지 않는다.** 위젯과 달리 사람이 보는 화면이라 필요하면 새로고침하면 된다.

### 붙이면서 터질 뻔한 것 — 로딩 표시가 편집기를 죽인다

`reloadCurrentOrg` 가 `loading = true` 를 세우는데, 조직 시간표 자동 저장 →
DB 변경 → Realtime 알림 → 리로드 가 매번 일어난다.
그대로 두면 **자동 저장할 때마다 화면이 "불러오는 중…"으로 갈아엎어지고,
편집기가 통째로 사라졌다 다시 뜨면서 입력이 날아간다.**

→ **로딩 표시는 그 조직을 '처음 열 때'만** 띄운다(`loadedOrgId` ref).
   이후의 재조회는 화면을 갈아엎지 않고 조용히 값만 바꾼다.
   (성공했을 때만 '읽었다'고 표시한다 — 실패했는데 표시하면 다음 재시도가 조용히 실패한다)

관리자가 편집 중일 때 자기 저장이 되돌아와 덮어쓰는 문제는 `OrgPlanEditor` 의 `dirty` 플래그가 막는다.

## 10. 초대 링크 + 관리자 승인 ✅

### 초대 링크
`?join=CODE` 를 실은 링크를 만들어 공유한다(`lib/inviteLink.ts`).
링크로 들어오면 참여 모달이 **코드가 채워진 채로** 열린다.

- **바로 가입시키지 않고 이름을 받는다.** 이름 없이 넣으면 겹쳐보기에서
  관리자가 누가 누군지 알 수 없다.
- 처리 후 **주소에서 코드를 지운다**(`history.replaceState`). 안 지우면 새로고침할 때마다
  모달이 다시 뜨고 신청이 또 나간다.
- 로그인 전이라도 안전하다 — 카카오 `redirectTo` 가 쿼리스트링을 그대로 살려서 돌아온다.

**카카오 SDK 를 쓰지 않았다.** 앱 키 발급·도메인 등록·스크립트 로딩이 붙는데,
폰에서 **Web Share** 를 띄우면 공유 목록에 카카오톡이 그대로 나온다 — 목적은 이미 달성된다.
PC 에서는 클립보드 복사로 떨어진다.

### 왜 승인이 필요한가
**링크는 아는 사람이면 누구나 쓸 수 있다.** 카톡방에 뿌린 링크가 밖으로 새면
모르는 사람이 팀 시간표를 본다. → 링크로 들어오면 '가입'이 아니라 **'가입 신청'**이 되고,
관리자가 승인해야 조직원이 된다.

**판정은 화면이 아니라 DB 가 한다.** `org_members.status` 를 두고
`is_org_member()` 가 **`active` 만 인정**한다. 승인 대기(`pending`)인 사람은
팀원 시간표도 조직 시간표도 **RLS 레벨에서 아예 못 읽는다.**

### 뚫릴 뻔한 것 — 셀프 승인
구성원 행의 update 정책에 `user_id = auth.uid()`(본인 수정 허용)가 있었다.
그대로 두면 **승인 대기자가 자기 `status` 를 `active` 로 바꿔 스스로를 승인**할 수 있다.
승인 절차가 통째로 무의미해진다.
→ 구성원 행 수정은 **관리자만**. 본인이 바꿔야 하는 건 이름뿐이므로
   그것만 `set_my_display_name` RPC 로 연다.

또 하나: `join_org` 의 `on conflict do nothing` 이 중요하다.
이미 `active` 인 사람이 링크를 다시 눌렀을 때 상태가 `pending` 으로 되돌아가면
**멀쩡한 조직원이 쫓겨난다.**

### 화면
- 신청자: "승인을 기다리는 중이에요" (빈 화면을 보여주면 고장 난 줄 안다)
- 관리자: 상단에 **참여 신청 N건** + 승인/거절 버튼 (아래에 묻히면 신청자가 하염없이 기다린다)
- 구성원 수·겹쳐보기 분모에서 **대기자는 세지 않는다** (실제보다 공유율이 나빠 보인다)

## 11. 다음 작업

### 위젯에 조직 일정 반영
바탕화면 위젯이 지금은 개인 일정만 보여준다. 배포된 **조직 시간표도** 보이게 한다.
- 위젯은 `useWidgetPresets`(읽기 전용)를 쓴다 — 조직도 같은 원칙(**절대 쓰지 않는다**)을 지킬 것
- 개인/조직 중 무엇을 볼지 고르는 방법이 필요하다(위젯은 UI 공간이 거의 없다 → 트레이 메뉴?)
- 오프라인 캐시(`widgetCache.ts`)도 조직 일정까지 확장해야 한다
