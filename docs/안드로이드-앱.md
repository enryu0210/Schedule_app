# 안드로이드 앱 (`mobile/`) — 1단계: 앱 셸

> 상태: **실기기 검증 완료** (2026-07-14, 갤럭시 Z Flip3 / Android 15).
> 앱 실행 → 카카오 로그인(커스텀탭 + 딥링크) → 시간표 표시까지 정상 동작 확인.
> 홈 위젯·상시 알림은 2단계.

---

## 1. 왜 Capacitor 인가 (설계 결정)

- **안드로이드 홈 위젯과 상시 알림은 웹 기술로 만들 수 없다.** 홈 위젯은 RemoteViews/Glance(코틀린),
  알림은 네이티브 서비스로만 가능하다. Tauri Android·PWA 를 써도 마찬가지다.
  → 네이티브 코드는 어차피 들어간다. 문제는 "얼마나 넣느냐"였다.
- **Capacitor 를 고른 이유**: 앱 본화면은 기존 React 웹앱을 그대로 재사용하고(로그인·동기화 코드를
  두 벌로 관리하지 않는다), 결과물이 **평범한 Android Studio 프로젝트**라 2단계에서 Glance 위젯을
  얹기 쉽다. (순수 코틀린 재구현은 작업량 3~4배, Tauri 는 모바일 플러그인 생태계가 빈약)

## 2. 데스크탑 위젯과 다른 점 — 화면을 "담는다"

| | 데스크탑 위젯(Tauri) | 안드로이드 앱(Capacitor) |
|---|---|---|
| 화면 출처 | 배포된 웹앱을 **URL 로 로드** | `web/dist` 를 **APK 안에 번들** |
| 웹 코드 수정 반영 | 푸시 → Vercel 배포하면 즉시 | **APK 를 다시 빌드·설치해야 함** |
| 인터넷 끊기면 | 껍데기부터 못 뜸 | 앱은 뜨고, 일정은 오프라인 캐시로 버팀 |

폰은 지하철·엘리베이터처럼 네트워크가 끊기는 곳에서 켜기 때문에 번들 방식을 골랐다.

## 3. 카카오 로그인 (웹과 다른 흐름)

웹은 현재 창을 카카오로 넘겼다 되돌아오면 끝이지만, 앱에서 그러면 **앱이 웹페이지로 변해버린다.**
게다가 인앱 웹뷰 로그인은 카카오가 제한할 수 있고, 폰에 이미 로그인된 카카오 세션도 못 쓴다.
그래서 표준 방식(크롬 커스텀탭 + 딥링크)을 쓴다 — `web/src/lib/nativeAuth.ts`

```
앱 → 로그인 URL 만 받아옴(skipBrowserRedirect)
  → 크롬 커스텀탭에서 카카오 로그인
  → Supabase 콜백 → ilgongbang://auth-callback#access_token=...
  → 앱이 딥링크 수신(App.appUrlOpen) → setSession → 커스텀탭 닫기
```

- 딥링크 주소: `ilgongbang://auth-callback` (AndroidManifest 의 intent-filter 와 한 글자도 달라선 안 됨)
- supabase-js 기본 흐름은 **implicit** 이라 토큰이 `#` 뒤에 온다. (PKCE 로 바꿀 때를 대비해 `?code=` 도 함께 처리)

### ⚠️ 반드시 해야 하는 수동 설정 (안 하면 로그인이 끝에서 막힌다)

Supabase 대시보드 → **Authentication → URL Configuration → Redirect URLs** 에 추가:

```
ilgongbang://auth-callback
```

## 4. 빌드 / 설치

**저장소 루트에서 PowerShell 로 실행한다.** 경로를 문서에 박아두지 않는 이유는 §4-1 참고.

```powershell
.\mobile\scripts\install-debug.ps1          # 빌드 + 폰에 설치
.\mobile\scripts\install-debug.ps1 -Sync    # 웹 코드를 고쳤을 때 (web/dist 재번들 후 설치)
```

스크립트가 폰 연결 여부를 먼저 확인하고(안 붙어 있으면 무엇을 켜야 하는지 알려준다),
JDK 와 adb 를 이 기기에서 찾아 잡은 뒤 `gradlew installDebug` 를 돌린다.

수동으로 하고 싶다면:

```powershell
. .\mobile\scripts\android-env.ps1   # $env:JAVA_HOME / adb 경로를 이 기기에 맞게 잡아준다
cd mobile; npm run sync              # 웹 코드를 고쳤으면 항상 이것부터
cd android; .\gradlew.bat assembleDebug   # 결과물: app/build/outputs/apk/debug/app-debug.apk
```

### 4-1. 경로를 하드코딩하지 않는 이유 (실제로 겪음)

문서에 `F:\dev\Schedule_app\...` 처럼 절대경로를 적어뒀다가 **PC 를 바꾸자(C: 드라이브) 복붙이 전부 실패**했다.
저장소 위치·Android Studio JDK 위치·adb 위치는 기기마다 다르다.
그래서 `mobile/scripts/android-env.ps1` 이 실행 시점에 **찾아내도록** 했다.

- 저장소 루트: `git rev-parse --show-toplevel` 로 물어본다
- JDK: Android Studio 내장 `jbr` 을 후보 경로에서 탐색 (Gradle 은 Java 23+ 와 안 맞아 이걸 써야 한다)
- adb: `%LOCALAPPDATA%\Android\Sdk\platform-tools` 등 후보 경로 + `ANDROID_HOME` 탐색

찾지 못하면 무엇을 설치/지정해야 하는지 알려주고 멈춘다.

### 4-2. 앱 아이콘

원본은 **SVG 한 장**(`assets/app-icon*.svg`)이고, 안드로이드가 요구하는 밀도별 PNG 는 **매번 다시 굽는다.**

```powershell
cd mobile; npm run icons     # SVG → PNG(1024) → mipmap-* 리소스 전부 생성
```

- 손으로 PNG 를 굽지 말 것. 밀도가 6종이라 한 벌만 옛 그림으로 남기 쉽다.
- 안드로이드 **적응형 아이콘**은 배경/전경을 따로 받아 런처가 원하는 모양(원·둥근사각…)으로 잘라낸다.
  그래서 원본이 3개다:
  - `app-icon.svg` — 완성본(모서리 둥근). 구형 런처용
  - `app-icon-background.svg` — 배경만. **모서리를 둥글게 깎지 않는다**(잘린 자리에 빈틈이 생긴다)
  - `app-icon-foreground.svg` — 카드만. 바깥이 잘려나가므로 **가운데로 0.9배 줄여** 안전 영역에 넣었다

### 4-3. 그 밖에

- 앱 ID: `com.ilgongbang.app` / 앱 이름: 일정공방
- 디버그 APK 는 디버그 키로 서명된다. "출처를 알 수 없는 앱" 설치 허용이 필요하다.
- 웹뷰 콘솔은 PC 크롬에서 `chrome://inspect` 로 볼 수 있다 (`webContentsDebuggingEnabled: true`).
- 새 PC 에서는 **`cd mobile; npm install` 을 먼저** 해야 한다. 안 하면 `cap` 이 없어
  `npm run sync` 가 "'cap' is not recognized" 로 죽는다(실제로 겪음).

## 5. 실기기 검증 결과 (2026-07-14, Z Flip3 / Android 15)

**핵심 흐름은 전부 통과했다.** 앱 실행 → 카카오 로그인(커스텀탭이 열리고 로그인 후 앱으로 복귀) →
클라우드에 있던 시간표 표시까지 정상. 딥링크(`ilgongbang://auth-callback`)도 제대로 되돌아왔다.

폰을 다시 연결할 때 절차:

1. **폰 준비** — 설정 → 휴대전화 정보 → **소프트웨어 정보** → 빌드번호 7번 탭(개발자 옵션) →
   **USB 디버깅 ON** → USB 연결 → 폰에 뜨는 "USB 디버깅을 허용하시겠습니까?" 허용
   (삼성은 '소프트웨어 정보' 한 단계 안에 빌드번호가 있다)
2. **설치** — 저장소 루트에서 `.\mobile\scripts\install-debug.ps1 -Sync`
3. 문제가 생기면 웹뷰 콘솔: PC 크롬 `chrome://inspect` → 연결된 기기의 앱 inspect

### 검증에서 드러난 것 (모두 수정 완료)

| 증상 | 진짜 원인 |
|---|---|
| 상단바가 상태바(시계·배터리) 밑에 깔림 | **안드로이드 15는 targetSdk 35 앱에 edge-to-edge 를 강제**한다. §7 참고 |
| 시간표를 스크롤하려는데 화면이 안 움직이고 일정이 끌려감 | `.wg-block` 에 `touch-action: none` 이 **항상** 걸려 있었다. 블록이 화면 대부분을 덮으니 사실상 매번 걸린다 |
| 상단바가 2줄로 부풀고 이름이 잘림 | 폰 가로폭에 요소가 너무 많았다 → 폰에서는 '공간' 라벨·위젯 내려받기·사용자 이름을 감춘다 |
| 앱 아이콘이 기본 아이콘 | 아이콘 리소스를 만든 적이 없었다 → §4-2 |

## 6. 모바일에서 편집은 "명시적으로" 켠다

시간표(그래프) 뷰는 이제 **기본이 보기 전용**이고, `✎ 편집` 버튼을 눌러야 고칠 수 있다.

폰에서는 화면을 넘기려고 블록에 손가락을 대는 것만으로 드래그가 시작돼 **일정이 조용히 옮겨졌다.**
"보다가 실수로 고쳐지는" 사고가 "고칠 때 버튼 한 번 더 누르는" 번거로움보다 훨씬 나쁘다.
편집 상태는 **일부러 기억하지 않는다** — 화면을 다시 열면 항상 잠긴 채로 시작한다.

## 7. 안드로이드 15 edge-to-edge 함정 (반드시 알 것)

`targetSdkVersion = 35` 인 앱은 안드로이드 15에서 **시스템이 강제로 edge-to-edge** 를 켠다.
웹뷰가 상태바 밑까지 파고들어 우리 상단바가 가려진다.

정석은 CSS 의 `env(safe-area-inset-top)` 이지만, **안드로이드 웹뷰의 safe-area 는 '노치(컷아웃)'만
반영하고 상태바 높이는 반영하지 않아** 기기마다 값이 다르다. 그래서 창 자체를 예전 동작으로 되돌렸다:

```xml
<!-- mobile/android/app/src/main/res/values/styles.xml -->
<item name="android:windowOptOutEdgeToEdgeEnforcement" tools:targetApi="35">true</item>
```

- **스플래시 테마(`AppTheme.NoActionBarLaunch`)에도 넣어야 한다.** 창은 그 테마로 만들어지기 때문.
- ⚠️ 이 플래그는 **과도기용**이다. `targetSdk` 를 36 으로 올리면 더는 먹지 않는다 →
  그때는 네이티브에서 `WindowInsets` 를 읽어 웹뷰에 여백을 주는 방식으로 바꿔야 한다.

## 8. 모바일 반응형 (실기기 화면을 보며 잡은 것)

폰 화면은 `adb shell screencap` 으로 캡처해 눈으로 확인하며 고쳤다. 추측으로 고치면 헛짚는다.

- **시간 축 압축** — 폰에서는 1시간 = 42px(데스크탑 52px). 52px 로는 06~18시밖에 안 보여
  "일주일을 한눈에" 라는 이 뷰의 존재 이유가 죽었다. CSS 로는 못 한다(블록 위치를 JS 가
  픽셀로 계산한다) → `useMediaQuery` 훅으로 폭을 JS 가 읽는다.
- **한글 줄바꿈** — `word-break: break-all` 이 한글을 단어 중간에서 끊었다("국가근/로(Argos").
  `keep-all` + `overflow-wrap: anywhere` 로 바꿔 단어를 지킨다.
- **겹친 블록은 계단식으로 쌓는다** — 예전엔 열 폭을 균등 분할(2개면 각 50%)했는데,
  폰은 열이 55px 라 한 칸이 27px 로 쪼그라들어 글자가 세로로 눌리고 누르기도 힘들었다.
  캘린더 앱들처럼 뒤 블록을 오른쪽으로 밀어 **겹쳐 쌓고** 각자 열 끝까지 폭을 갖는다(→ 36px).
  - 인라인 `style` 이 CSS 를 이기므로, **드래그 중 z-index 는 인라인에서 올려야** 다른 블록 뒤로 안 숨는다.
- **탭 → 상세 보기(`BlockDetail`)** — 열이 7개인 이상 이름이 잘리는 건 물리적 한계다.
  잘린 이름은 눌러서 읽는다. 상세에서 '편집'을 누르면 편집 모드를 켜고 편집 모달로 넘어간다.
