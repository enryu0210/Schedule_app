# 구글 캘린더 연동 (백그라운드 자동 동기화)

> 목표: 사용자의 구글 캘린더 일정을 **주기적으로 자동으로** 가져와, 앱 안에 **"달(월) 일정"** 으로 보여준다.
> 주간 프리셋(반복 시간표)과 달리, 캘린더는 **날짜에 못박힌 등록**이라 구조가 다르다.
> 나중에 AI 가 이 달 일정을 유동적으로 주간 시간표에 배치할 예정 → 지금은 **'받아서 달 뷰로 보여주기'** 까지가 범위.

---

## 0. 왜 이렇게 설계하는가 (핵심 결정)

### 0.1 왜 '달' 구조인가
- 기존 `Preset` 은 **월~일 7일이 반복**되는 주간 시간표다(방학/학기중 같은 '상황' 프리셋).
- 구글 캘린더 일정은 "7월 23일 14시 회의"처럼 **특정 날짜**에 붙는다. 주간 반복 틀에 그대로 못 넣는다.
- 그래서 **개인/조직 데이터와 마찬가지로 테이블·타입부터 분리**한다. 주간 프리셋을 건드리지 않는다.

### 0.2 왜 refresh 토큰을 '서버(Edge Function)'에만 두는가 (보안)
- 백그라운드 자동 동기화는 **앱이 꺼져 있어도** 구글에 접근해야 한다 → 장기 토큰(refresh token)이 필요하다.
- 이 토큰을 **클라이언트(localStorage)에 두면 절대 안 된다.** 보안 감사(2026-07-22)에서 이미 경고:
  안드로이드 WebView localStorage 는 여러 경로로 유출될 수 있다(디버깅/백업).
  refresh 토큰이 새면 **그 사람 캘린더를 무기한 읽힌다.**
- 따라서 **OAuth 코드 교환과 토큰 저장은 전부 Supabase Edge Function(서버)** 에서 한다.
  클라이언트는 refresh 토큰도, 구글 client_secret 도 **절대 만지지 않는다.**
- 스코프는 **읽기 전용(`calendar.readonly`)** 만. 우리는 일정을 '읽어서 보여줄' 뿐, 쓰지 않는다.

### 0.3 왜 서버 사이드 '코드 플로우'인가 (Supabase 구글 provider 를 안 쓰는 이유)
- Supabase 의 구글 provider 로그인도 `provider_refresh_token` 을 주긴 하지만, 그 토큰이
  **세션 객체를 통해 클라이언트로 내려온다** → 0.2 원칙에 어긋난다.
- 게다가 이 앱의 주 로그인은 **카카오**다. 캘린더 접근만을 위해 로그인 방식을 흔들 이유가 없다.
- 그래서 캘린더 접근은 **카카오 로그인과 별개의, 서버가 주도하는 구글 OAuth** 로 붙인다.
  (사용자 눈엔 "구글 캘린더 연결" 버튼 하나)

## 1. 아키텍처 한눈에

```
[웹/위젯 클라이언트]                    [Supabase]                         [구글]
  "구글 캘린더 연결" 클릭
      │  (state = 내 세션 증명)
      └──── 구글 동의화면 열기 ─────────────────────────────────────────▶ 로그인+동의(calendar.readonly, offline)
                                                                                │  ?code=...
      ┌───────────────── redirect ──────── Edge Fn: google-calendar-callback ◀──┘
      │                                        │ code + client_secret → 토큰 교환
      │                                        │ refresh_token 을 google_calendar_tokens 에 저장(서버 전용)
      │                                        ▼
      │                                  [google_calendar_tokens]  ← 클라이언트 접근 전면 차단(RLS)
      │
      │   (주기적으로, 앱과 무관하게)
      │                                  Edge Fn: google-calendar-sync (스케줄)
      │                                        │ refresh_token → access_token → Calendar API events.list
      │                                        │ 이벤트를 달 구조로 변환
      │                                        ▼
      │                                  [calendar_schedules]  ← 본인만 읽기(RLS) + Realtime 발행
      └──── Realtime 구독 ◀──────────────────┘
   달 뷰에 표시
```

핵심: **토큰은 서버 안에서만 흐르고, 클라이언트는 결과(달 일정)만 읽는다.**

## 2. 데이터 모델

### 2.1 클라이언트 타입 (`web/src/types.ts`) — Phase 1 에서 추가
```ts
// 구글 캘린더 이벤트 하나 (날짜에 못박힌 일정)
interface CalendarEvent {
  id: string;        // 구글 이벤트 id (동기화 때 중복/갱신 판별)
  date: string;      // "YYYY-MM-DD" (시작일 기준)
  start: string;     // "HH:MM" (종일 일정이면 "00:00")
  end: string;       // "HH:MM"
  label: string;     // 이벤트 제목
  allDay: boolean;   // 종일 일정 여부
}

// 가져온 캘린더 전체 (여러 날/달에 걸친 이벤트 목록)
interface CalendarSchedule {
  events: CalendarEvent[]; // 날짜순 정렬 가정
  rangeStart: string;      // 동기화한 구간 시작 "YYYY-MM-DD"
  rangeEnd: string;        // 동기화한 구간 끝
  syncedAt: number;        // 마지막 동기화 시각(ms) — "○분 전 동기화" 표시용
}
```

### 2.2 DB 테이블 (`supabase/google-calendar-schema.sql`) — Phase 2 에서 작성·**먼저 실행**
```sql
-- (1) 구글 토큰: 서버(Edge Function, service_role)만 접근. 클라이언트는 한 줄도 못 읽는다.
create table public.google_calendar_tokens (
  user_id       uuid primary key references auth.users(id) on delete cascade,
  refresh_token text not null,        -- ⚠ 서버 전용. 클라이언트 노출 금지.
  connected_at  timestamptz not null default now()
);
alter table public.google_calendar_tokens enable row level security;
-- 정책을 '하나도' 만들지 않는다 → RLS 가 모든 클라이언트 접근을 기본 거부.
--   Edge Function 은 service_role 로 붙어 RLS 를 우회하므로 정상 동작한다.

-- (2) 동기화된 달 일정: 본인만 읽기. 쓰기는 Edge Function(service_role)만.
create table public.calendar_schedules (
  user_id     uuid primary key references auth.users(id) on delete cascade,
  schedule    jsonb not null,          -- CalendarSchedule 구조
  synced_at   timestamptz not null default now()
);
alter table public.calendar_schedules enable row level security;
create policy "본인 캘린더 조회" on public.calendar_schedules
  for select using (auth.uid() = user_id);
-- insert/update 정책은 만들지 않는다 → 클라이언트는 못 쓰고, Edge Function 만 쓴다.

-- (3) 위젯/앱이 즉시 갱신되도록 Realtime 발행
do $$ begin
  alter publication supabase_realtime add table public.calendar_schedules;
exception when duplicate_object then null; end $$;
```
> 프로젝트 규칙: **이 SQL 을 Supabase 에서 먼저 실행→확인한 뒤** 관련 클라이언트 코드를 배포한다.
> (없는 테이블을 조회하면 화면이 통째로 깨진 전례가 있음)

## 3. 단계별 계획 (Phase)

### Phase 0 — 구글 클라우드 설정 **(← 당신이 할 일, §5 체크리스트)**
- 이게 끝나야 Phase 2 이후가 돌아간다. Phase 1(클라이언트 뼈대)은 이것과 **병렬로** 진행 가능.

### Phase 1 — 클라이언트 달(월) 뷰 + 타입 (구글 없이도 만드는 부분)
- `types.ts` 에 `CalendarEvent` / `CalendarSchedule` 추가
- **월 뷰 컴포넌트**(`MonthView.tsx`): 달력 격자에 날짜별 이벤트 표시. 목(mock) 데이터로 먼저 완성.
- "구글 캘린더 연결" 버튼 자리(아직 동작은 stub)
- 빌드 통과로 1차 검증 (인증 게이트 때문에 브라우저 QA 제한)

### Phase 2 — OAuth 연결 (토큰 확보·저장)
- `supabase/google-calendar-schema.sql` 작성 → **Supabase 에서 실행** → 확인
- Edge Function `google-calendar-callback`: `?code=` → 토큰 교환 → `google_calendar_tokens` 저장
- 클라이언트 "연결" 버튼 → 구글 동의화면 열기(state 로 본인 증명, `access_type=offline&prompt=consent`)
- 연결 성공/실패 화면 처리

### Phase 3 — 백그라운드 동기화
- Edge Function `google-calendar-sync`: 토큰 있는 사용자별로 access_token 재발급 →
  Calendar API `events.list`(이번 달~다음 달) → 달 구조로 변환 → `calendar_schedules` 갱신
- 스케줄 실행(Supabase Scheduled Functions / pg_cron + pg_net) — 예: 6시간마다
- 클라이언트: `calendar_schedules` 를 Realtime 구독 → 달 뷰 자동 갱신
- "지금 동기화" 수동 버튼(선택)

### Phase 4 — (나중) AI 로 달 일정 → 주간 시간표 유동 배치
- 로드맵. 이번 범위 아님. 캘린더 이벤트 제목/설명은 **외부 입력**이므로
  LLM 시스템 프롬프트에 그대로 넣지 말 것(프롬프트 인젝션) — 사용자 메시지 위치에 격리.

## 4. 보안 체크리스트 (감사 2026-07-22 연장선)
- [ ] refresh 토큰·client_secret 은 **Edge Function(서버)** 에만. 클라이언트 응답·로그에 절대 싣지 않는다.
- [ ] `google_calendar_tokens` 는 RLS 로 **클라이언트 접근 전면 차단**(정책 0개 + service_role 만 사용).
- [ ] 스코프는 `calendar.readonly` 만. 더 넓은 권한 요청 금지.
- [ ] OAuth `state` 로 CSRF 방지 + 어떤 사용자의 연결인지 서버에서 검증.
- [ ] client_secret 은 Supabase Edge Function 시크릿(`supabase secrets set`)으로. 저장소·`.env`(클라) 금지.
- [ ] 연결 해제 시 `google_calendar_tokens` 행 삭제 + 구글에서 토큰 revoke.

## 5. 당신이 준비할 것 — 구글 클라우드 설정 (Phase 0)

> 아래를 마치면 **client_id(공개)** 와 **client_secret(비밀)** 이 나온다. 그 두 값을 알려주면(secret 은 안전히) 서버 연동을 붙인다.

1. **Google Cloud Console** (https://console.cloud.google.com) → 프로젝트 생성(예: `ilgongbang`).
2. **API 라이브러리** → **Google Calendar API** 검색 → **사용 설정(Enable)**.
3. **OAuth 동의 화면(OAuth consent screen)**:
   - User Type: **External**
   - 앱 이름/지원 이메일 입력
   - **스코프 추가**: `.../auth/calendar.readonly` (Google Calendar API, read-only)
   - **테스트 사용자**에 본인 구글 계정 추가 (게시 전에는 테스트 사용자만 연결 가능)
4. **사용자 인증 정보(Credentials)** → **OAuth 2.0 클라이언트 ID 만들기**:
   - 유형: **웹 애플리케이션**
   - **승인된 리디렉션 URI**: Supabase Edge Function 콜백 주소
     `https://<프로젝트ref>.supabase.co/functions/v1/google-calendar-callback`
     (실제 `<프로젝트ref>` 는 Supabase 대시보드 URL 에서 확인 — 붙일 때 함께 맞춘다)
5. 나온 **client_id / client_secret** 확보. (client_secret 은 절대 화면 캡처·깃 커밋 금지)

> ⚠️ 게시(Publishing) 심사: 앱을 "테스트" 상태로 두면 등록한 테스트 사용자만 연결된다.
> 본인·소수만 쓸 거면 테스트 상태로 충분. 불특정 다수 대상이면 구글 검증(verification)이 필요하다.
