## 0.5 돈($99) 결정은 미룰 수 있다 — 무료로 1단계까지 간다 (2026-07-15 확인)

**$99 를 지금 결정하지 않아도 iOS 를 검증할 수 있다.** 무료 Apple ID 만으로도:
- `cap add ios` → Xcode 로 시뮬레이터/내 아이폰에서 **앱 셸 + 카카오 로그인 + 시간표까지(1단계 전부)** 확인 가능.

$99 가 **실제로 필요해지는 시점**은 그 다음이다:
- **위젯**(App Group 권한) — 3단계 / **TestFlight 지인 배포** — 4단계 / 7일 만료 없는 상시 사용.

→ 순서: **무료로 1단계 검증 → 위젯·배포가 마음에 들면 그때 $99 결제.**
   진짜 병목은 돈이 아니라 **Mac 한 대**다(§2, 지인 Mac 하루 또는 public 저장소라 GitHub Actions 무료).

> 참고: §10 에서 걱정한 "안드로이드 홈 위젯이 미완이라 계산 로직이 3벌 된다"는 **해소됨** —
> 안드로이드 홈 위젯은 v0.2.0/v0.2.1 로 완료(`docs/프로젝트-현황.md`). iOS 로 가도 순서 문제 없음.

## 1. 결론부터 — iOS 는 "안드로이드 한 번 더"가 아니다

| | 안드로이드 (완료) | iOS (계획) |
|---|---|---|
| 앱 본화면 | `web/dist` 번들 | **그대로 재사용** ✅ |
| 카카오 로그인 | 커스텀탭 + 딥링크 | **코드 그대로** (`nativeAuth.ts` 는 이미 플랫폼 중립) ✅ |
| "지금 하는 일정" 상시 표시 | 지워지지 않는 알림 | **불가능** → 잠금화면 위젯으로 대체 ❌ |
| 위젯 화면 | Kotlin + Glance | **Swift + SwiftUI(WidgetKit)** — 새로 작성 |
| 빌드 | 윈도우에서 됨 | **macOS + Xcode 필수** |
| 배포 | APK 파일 전달 | **TestFlight** (파일 전달 불가) |

**웹 화면과 로그인은 공짜로 따라오고, 네이티브 부분은 통째로 다시 짠다**고 보면 된다.
안드로이드에서 만든 `ScheduleStore`(지금/다음 일정 계산)의 **로직은 재사용 가치가 있지만
언어가 달라 코드는 못 옮긴다** — Java → Swift 로 옮겨 적어야 한다.

## 2. 가장 먼저 풀어야 할 문제 — Mac 이 없다

**iOS 앱은 윈도우에서 빌드할 수 없다.** Xcode 가 macOS 전용이고, 대안(우회 툴체인)은 없다.
Capacitor 를 쓰든 뭘 쓰든 마지막에 `xcodebuild` 는 반드시 Mac 에서 돈다.

선택지는 셋이다.

### (A) GitHub Actions macOS 러너 — **이 저장소는 public 이라 무료** ⭐ 유력

공개 저장소는 GitHub 이 제공하는 **macOS 러너를 무료로** 쓸 수 있다(private 이면 분당 과금, macOS 는 리눅스의 10배).
`.github/workflows/ios.yml` 에서 `runs-on: macos-latest` 로 빌드하면 **Mac 을 사지 않고 IPA 를 굽는다.**

- 장점: 추가 비용 0원. 릴리스 자동화까지 그대로 이어진다.
- 단점: **화면을 볼 수 없다.** 시뮬레이터를 띄워 눈으로 확인하거나, 웹뷰 콘솔을 열어 디버깅하는 게 불가능하다.
  빌드가 깨지면 로그만 보고 고쳐야 한다. 서명 인증서를 CI 에 넣는 준비(§7)가 처음 한 번 꽤 번거롭다.
- **현실적인 판단**: 첫 셋업까지는 Mac 앞에 앉는 편이 훨씬 빠르다. 한 번 굴러가기 시작하면 CI 가 좋다.

### (B) 지인/카페의 Mac 을 하루 빌린다

- 첫 셋업(`cap add ios` → Xcode 로 열기 → 서명 → 실기기 설치)을 **하루 안에 끝낼 수 있다.**
- 그 뒤 웹 화면만 고치는 작업은 **Mac 없이도 CI(A)로 배포**할 수 있다.
- 비용 0원. **가장 현실적인 시작점.**

### (C) 클라우드 Mac (MacinCloud, MacStadium 등)

- 월 $20~30 수준. 원격 데스크톱으로 Xcode 를 쓴다.
- 실기기 연결이 안 되므로 **시뮬레이터까지만.** 내 아이폰에 꽂아 테스트하려면 결국 실물 Mac 이 필요하다
  (TestFlight 로 우회 가능하지만 왕복이 느리다).

> **권장**: (B)로 첫 셋업 → (A)로 이후 빌드 자동화. 중고 Mac mini 구매는 iOS 를 계속 갈 확신이 선 뒤에.

## 4. ⚠️ 안드로이드의 "상시 알림"은 iOS 에 없다 (설계 변경 필요)

안드로이드에서는 지워지지 않는 ongoing 알림으로 "지금 하는 일정"을 항상 띄웠다(`docs/안드로이드-앱.md` §9).
**iOS 는 이런 알림을 원천적으로 허용하지 않는다.** 모든 알림은 사용자가 밀어서 지울 수 있고,
앱이 알림을 계속 붙들고 있을 방법이 없다. 우회로도 없다(정책이 아니라 API 가 없다).

대안은 둘이며, **잠금화면 위젯을 먼저** 한다.

### (1) 잠금화면 / 홈화면 위젯 (WidgetKit) ⭐ 채택

- iOS 16+ 잠금화면 위젯(`accessoryRectangular`)에 **"지금: 국가근로 · 2시간 10분 남음"** 을 띄운다.
- 잠금화면은 하루에 수십 번 보는 화면이라 **"수시로 확인한다"는 원래 목표에 가장 잘 맞는다.**
- 홈화면 위젯(`systemSmall`/`systemMedium`)도 같은 코드로 함께 나온다.

### (2) Live Activity (다이나믹 아일랜드) — 나중에

- 배달앱처럼 잠금화면 상단·다이나믹 아일랜드에 진행률을 띄운다. 가장 "상시 알림"에 가깝다.
- **하지만 최대 8시간 제한**이 있어 하루 종일 유지할 수 없다. 일정 하나가 시작될 때 띄우고
  끝나면 내리는 식이라면 쓸 만하지만, 안드로이드처럼 "항상 떠 있는" 것은 불가능하다.
- 위젯이 먼저 돌아간 뒤에 얹는다.

## 5. 위젯 설계 — 데이터는 앱이 넘겨주고, 위젯은 그리기만

**위젯이 Supabase 를 직접 부르게 하면 안 된다.** 인증(카카오 세션)을 위젯 안에서 또 만들어야 하고,
로그인 상태가 앱과 어긋난다. 안드로이드에서 세운 원칙을 그대로 지킨다.

```
React 앱(웹뷰)            Capacitor 다리          Swift
─────────────────────────────────────────────────────────────
usePresetStore
  → 주간 시간표    ──▶  SchedulePlugin.swift  ──▶  App Group UserDefaults
                                                    (group.com.ilgongbang.app)
                                                          │
                                                          ▼
                                                    WidgetKit Extension
                                                    (지금/다음 일정 계산 → SwiftUI 로 그림)
```

- **App Group** 은 앱과 위젯이 같은 저장소를 보게 해주는 iOS 장치다. **유료 계정에서만 켤 수 있다**(§3).
- 웹이 넘기는 JSON 형식은 **안드로이드에서 쓰던 것을 그대로 재사용한다** (`web/src/lib/scheduleNotice.ts` 의
  `toNativeWeek`): `[[{ s: "07:30", e: "17:00", l: "국가근로" }, …], …]` (0=월 … 6=일).
  → 웹 코드는 **거의 안 고쳐도 된다.** `isNoticeSupported()` 가 `platform === "android"` 로 막고 있으니
  이 게이트만 iOS 를 포함하도록 넓히고, 플러그인 이름을 양쪽에서 맞춰주면 된다.

### 위젯 갱신은 "미리 타임라인을 깔아두는" 방식

안드로이드는 `AlarmManager` 로 일정 경계마다 깨워서 알림을 다시 그렸다.
iOS 는 **앱이 위젯을 마음대로 깨울 수 없다.** 대신 `TimelineProvider` 가
**"몇 시에 화면이 어떻게 바뀔지"를 미리 배열로 제출**하면 시스템이 그 시각에 알아서 갈아 끼운다.

- 우리 데이터는 **미래가 이미 정해져 있다**(주간 시간표) → 이 방식과 궁합이 완벽하다.
  오늘의 모든 일정 시작/종료 시각을 엔트리로 만들어 한 번에 제출하면 끝.
- 미리 제출한 엔트리는 **갱신 예산(하루 40~70회)을 소모하지 않는다.** 반면
  `WidgetCenter.reloadTimelines()` 를 남발하면 예산이 말라 위젯이 멈춘다 → **시간표가 바뀐 순간에만** 부른다.

## 6. 카카오 로그인 — 거의 그대로 된다 ✅

`web/src/lib/nativeAuth.ts` 는 이미 `Capacitor.isNativePlatform()` 으로만 판별하고 안드로이드를 특정하지 않는다.
iOS 에서 해야 할 일은 **네이티브 설정 3줄뿐**이다.

1. `ios/App/App/Info.plist` 에 URL 스킴 등록 (안드로이드의 intent-filter 에 해당)
   → `ilgongbang` — **한 글자도 다르면 안 된다.**
2. Supabase Redirect URLs: **이미 등록된 `ilgongbang://auth-callback` 를 그대로 쓴다** (추가 작업 없음)
3. 카카오 개발자 콘솔: iOS 플랫폼 등록 (번들 ID `com.ilgongbang.app`)

> iOS 는 `Browser.open()` 이 **SFSafariViewController** 로 열린다(안드로이드의 커스텀탭에 해당).
> 사파리에 로그인된 카카오 세션을 그대로 쓸 수 있어 동작이 오히려 더 매끄럽다.

## 7. 배포 — 파일로 못 준다, TestFlight 를 쓴다

안드로이드는 APK 파일을 카톡으로 보내면 끝이었다. **iOS 는 그게 안 된다.**

- **TestFlight**: 테스터를 초대하면 TestFlight 앱에서 받아 깐다. 지인 배포는 이게 정답이다.
  - 내부 테스터(App Store Connect 계정 보유자) 최대 100명 — 심사 없이 즉시
  - 외부 테스터(이메일/링크 초대) 최대 10,000명 — **첫 빌드만 간단한 베타 심사**(보통 하루)
  - ⚠️ **빌드는 90일 뒤 만료**된다. 계속 쓰게 하려면 주기적으로 새 빌드를 올려야 한다.
- Ad-hoc(기기 UDID 100대 등록 후 IPA 직접 설치)도 가능하지만, 지인마다 UDID 를 받아야 해서 더 번거롭다.

### CI 서명 (§2-A 로 갈 때)

인증서·프로비저닝 프로파일을 GitHub Secrets 에 넣어야 한다. `fastlane match` 가 표준.
**서명 키를 잃으면 기존 사용자에게 업데이트를 못 준다** — 위젯 서명 키(`~/.tauri/`), 안드로이드 키스토어
(`~/.ilgongbang/`)와 **똑같은 함정**이다. 백업 위치를 한곳에 모아 관리할 것.

- [ ] **4단계 — TestFlight** 로 지인 배포
- [ ] **5단계(선택) — Live Activity** (§4-2)
- [ ] **6단계(선택) — GitHub Actions 자동 빌드** (§2-A)

## 9. 미리 알아둘 함정 (안드로이드에서 겪은 것의 iOS 판)

- **`cap add ios` 는 윈도우에서도 폴더는 만들지만 `pod install` 에서 죽는다**(CocoaPods 가 macOS 전용).
  → **Mac 에서 처음부터 하는 게 맞다.** 윈도우에서 만든 반쪽 폴더를 커밋해두면 나중에 꼬인다.
- **웹 코드를 고치면 `npm run sync` 후 재빌드**해야 반영된다 — 안드로이드와 똑같다
  (`web/dist` 를 앱 안에 번들하므로). 데스크탑 위젯처럼 "웹만 배포하면 끝"이 아니다.
- **위젯은 웹뷰가 아니다.** 시간표 화면을 위젯에 그대로 띄울 수 없고, SwiftUI 로 다시 그려야 한다.
  (안드로이드 Glance 와 같은 제약)
- **`versionCode` 에 해당하는 것은 `CFBundleVersion`** 이다. TestFlight 는 같은 빌드 번호를 두 번 받지 않는다.
- 무료 계정으로 시작했다가 유료로 바꾸면 **번들 ID·서명이 달라져 앱을 지우고 다시 깔아야 할 수 있다.**
  → 처음부터 유료로 시작하는 편이 낫다.
