# 위젯 설계 (데스크탑 우선)

> 작성일: 2026-07-13
> 목표: "수시로 확인할 수 있는 위젯" — 지금 무슨 일정을 하고 있는지 항상 보이게 한다.

---

## 1. 요구사항 (원 스펙)

- **데스크탑(윈도우 우선)**: 바탕화면 위에 띄워놓는 형태. **Win+D를 눌러도 내려가지 않아야 함.**
- **안드로이드**: 홈 화면 위젯 + 네비게이션/알림 바에서 현재 진행 중인 일정 확인.

---

## 2. 기술 선택 — 왜 Tauri인가

| | Tauri (선택) | Electron |
|---|---|---|
| 상주 메모리 | 30~60MB (윈도우 내장 WebView2 사용) | 150~200MB (Chromium 통째로 탑재) |
| 배포 파일 | ~10MB | ~150MB |
| Win32 API 호출 | Rust에서 정식 지원(`windows` 크레이트) | `ffi-napi`/`koffi` 같은 FFI 우회 필요 |
| 설치 부담 | Rust + VS 빌드 도구 필요(일회성) | Node만 있으면 됨 |

**위젯은 하루 종일 켜져 있는 프로그램**이다. 상주 메모리가 결정적이라 Tauri를 택했다.

---

## 3. Win+D를 이겨내는 방법

일반적인 "항상 위(always-on-top)" 창은 Win+D(바탕화면 보기)에 **같이 내려간다.** 두 가지 해법이 있다.

1. **WorkerW 자식으로 붙이기** — 창을 바탕화면 레이어의 자식으로 만들어 "바탕화면의 일부"가 되게 한다.
   Win+D의 영향을 아예 안 받지만 **바탕화면 아이콘 뒤에 깔린다.**
2. **`WM_WINDOWPOSCHANGING` 가로채기** — Win+D는 창을 화면 밖(-32000, -32000)으로 밀어내는 방식으로
   동작한다. 이 메시지를 가로채 그 이동을 무시하면 창이 그대로 남는다. **아이콘 위에 뜬다.** ← **채택**

> 기존 라이브러리(electget, tauri-plugin-wallpaper)는 모두 취미 수준이고,
> tauri-plugin-wallpaper 저자조차 "취미가 아니면 Win32 로직을 직접 구현하라"고 권고한다.
> → **Win32 호출은 우리가 직접 작성한다.**

---

## 4. 구조 — 데이터는 절대 두 벌로 관리하지 않는다

순진하게 접근하면 데스크탑 앱에 **카카오 로그인 + Supabase 클라이언트 + 동기화 로직을 새로 붙이게 된다.**
같은 코드를 두 벌 관리하는 셈이고, 버그가 두 배로 생긴다.

**대신 Tauri 창이 이미 배포된 웹앱을 그대로 로드한다.**

```
┌─ Tauri (Rust) — "창의 성질" ──────────────┐
│  · 테두리 없는 투명 창                      │
│  · 항상 위 + Win+D 무시 (Win32)            │
│  · 드래그로 이동, 트레이 아이콘             │
│                                            │
│   └─ WebView2 ─────────────────────────┐  │
│      https://<배포주소>/?widget=1       │  │
│      → 로그인 세션·데이터 동기화가        │  │
│         웹앱 그대로 공짜로 따라옴          │  │
│      → 우리가 만들 건 WidgetView 화면뿐   │  │
│      ────────────────────────────────┘  │
└────────────────────────────────────────────┘
```

- **웹 쪽 할 일**: `?widget=1` 일 때 렌더링되는 컴팩트한 `WidgetView` (← 1단계, Rust 없이 가능)
- **Rust 쪽 할 일**: 창의 성질 + Win32 (← 2단계, Rust 설치 후)

**트레이드오프**: 오프라인이면 빈 화면이 된다. 추후 로컬 캐시로 보완한다.

---

## 5. WidgetView 화면 사양

- 지금 진행 중인 일정을 **크게** (없으면 "지금은 일정이 없어요")
- 다음 일정 2개를 작게
- 현재 시각 / 오늘 요일
- 데이터는 **읽기 전용** — 위젯은 절대 클라우드에 쓰지 않는다.
  (웹에서 수정한 내용을 반영하려고 60초마다 다시 읽기만 한다)
- 비로그인 상태면 로그인 안내만 노출

## 6. 단계별 계획

- [x] 1단계 — 웹앱에 `?widget=1` WidgetView 추가 + 배포 (브라우저에서 바로 확인 가능)
- [x] 2단계 — Tauri 셸: 프레임리스/투명 창이 웹앱 위젯 URL을 로드 (`desktop/`)
- [x] 3단계 — Win32: `WM_WINDOWPOSCHANGING` 가로채 Win+D 무시 (`src-tauri/src/win_desktop.rs`)
- [x] 4단계 — 트레이 아이콘, 위치 기억, 시작 프로그램 등록 (`src-tauri/src/tray.rs`)
- [ ] 5단계 — 오프라인 캐시
- [ ] 이후 — 안드로이드 위젯 + 알림바 상시 표시

### 4단계에서 배운 것

- **"항상 위"는 기본으로 끈다.** 처음엔 켜뒀더니 브라우저·게임 위까지 위젯이 떠서 방해가 됐다.
  Win+D 무시는 Win32 쪽(3단계)에서 따로 하므로, 항상 위가 꺼져 있어도
  **"바탕화면 보기를 하면 위젯만 남는다"** 는 원래 목표는 그대로 달성된다.
  → 필요할 때만 트레이 메뉴에서 켠다.
- **위젯 창은 결국 웹뷰라, 손대지 않으면 웹페이지 티가 난다.** 실제로 겪은 것:
  우클릭 → 브라우저 컨텍스트 메뉴 / 상단 더블클릭 → 창 최대화 / 드래그하면 글자 선택.
  → `maximizable: false` + `contextmenu` 차단 + `user-select: none` 으로 막았다.
- **위젯 창에서 로그인하면 위젯이 전체 앱 화면으로 바뀌던 버그.**
  카카오 OAuth 리다이렉트가 `window.location.origin` 으로 돌아오면서 `?widget=1` 이 날아갔다.
  → 돌아올 주소에 쿼리를 살리고, 안전망으로 위젯 모드를 `sessionStorage` 에도 기억시킨다
    (`web/src/lib/widgetMode.ts`).
- **위치 기억·자동 실행은 직접 만들지 않는다.** 공식 플러그인(`tauri-plugin-window-state`,
  `tauri-plugin-autostart`)을 쓴다. 단 window-state 는 **위치/크기만** 복원한다 —
  표시 여부까지 복원하면 "숨긴 채로 종료 → 다음 실행에도 안 보임"이 되어 고장으로 오해한다.
- **트레이는 부가기능이 아니라 필수다.** 위젯 창엔 테두리도 작업표시줄 버튼도 없어서,
  트레이가 없으면 사용자가 앱을 끌 방법이 아예 없다.
- **앱 이름은 한글로.** `Schedule Widget` 이라 시작 메뉴에서 "일정공방"으로 검색해도 안 나왔다.
  트레이에서 종료하면 시작 메뉴로 다시 켜야 하는데, 이름으로 못 찾으면 사실상 다시 못 켠다.
- **위젯 안에서 로그인하려면 창을 키워야 한다.** 카카오 로그인 페이지는 300x230 에 안 들어간다.
  "브라우저에서 로그인하세요"로 우회할 수 없다 — 브라우저와 위젯 웹뷰는 **저장소가 따로**라
  브라우저에서 로그인해도 위젯엔 세션이 안 생긴다.
  → 로그인 버튼을 누르면 창을 420x640 으로 키우고, 로그인 왕복이 끝나면 원래 크기로 되돌린다
    (`web/src/lib/widgetWindow.ts`). 원래 크기는 **localStorage** 에 적어둔다 —
    로그인 도중 위젯을 꺼버리면 window-state 플러그인이 "커진 크기"를 저장해버리기 때문에,
    다음 실행에서 되돌릴 근거가 남아 있어야 한다.

## 7. 폴더 구조

```
desktop/
├─ package.json          # @tauri-apps/cli 만 (프론트엔드 빌드 없음)
├─ public/index.html     # 자리표시용 (실제 화면은 배포된 웹앱을 로드)
└─ src-tauri/
   ├─ tauri.conf.json    # 창 속성 + 위젯 URL
   ├─ capabilities/      # 원격 URL(배포 웹앱)에 창 이동 권한 부여
   └─ src/
      ├─ lib.rs          # 앱 부팅 + 플러그인(위치 기억/자동 실행) 등록
      ├─ tray.rs         # 트레이 아이콘 + 메뉴 (보이기·항상 위·자동 실행·종료)
      └─ win_desktop.rs  # Win+D 무시 (서브클래싱)
```

## 8. 사전 준비 (개발 환경)

```
winget install --id Rustlang.Rustup -e
winget install --id Microsoft.VisualStudio.2022.BuildTools -e --override "--wait --quiet --add Microsoft.VisualStudio.Workload.VCTools --includeRecommended"
```

**함정 2가지 (실제로 겪음):**
1. **rustup은 VS C++ 빌드 도구를 자동으로 깔아주지 않는다.** Tauri는 MSVC 링커가 필수라
   위 두 번째 명령을 반드시 따로 실행해야 한다.
2. **Git Bash / MSYS 에서 `cargo build` 를 하면 링크가 깨진다.** `C:\MinGW\msys\1.0\bin\link.exe`
   (GNU coreutils의 link)가 MSVC `link.exe` 를 가려버리기 때문이다.
   → **PowerShell 에서 빌드할 것.**

## 9. 빌드 / 실행

```powershell
cd desktop
npm run dev     # 개발 모드로 위젯 창 띄우기
npm run build   # 설치 파일(NSIS) 생성 → src-tauri/target/release/bundle/nsis/
```

## 10. 릴리스 (웹에서 내려받게 하기)

설치 파일은 **저장소에 넣지 않는다.** 수 MB 짜리 바이너리를 버전마다 커밋하면 저장소가 무거워지고,
새 위젯을 내보낼 때마다 웹을 재배포해야 한다. → **GitHub Releases** 에 올린다.

```powershell
cd desktop
npm run build

# 자산 이름에 버전을 넣지 않는 것이 핵심 —
# 그래야 웹의 "latest/download/<고정이름>" 링크가 영원히 최신 파일을 가리킨다.
copy "src-tauri\target\release\bundle\nsis\Schedule Widget_0.1.0_x64-setup.exe" "$env:TEMP\Schedule-Widget-Setup.exe"
gh release create widget-v0.1.0 "$env:TEMP\Schedule-Widget-Setup.exe" --title "..." --notes "..."
```

- 웹의 다운로드 버튼: `web/src/components/DownloadWidget.tsx` (상단바, 모바일에서는 숨김)
- 고정 주소: `https://github.com/enryu0210/Schedule_app/releases/latest/download/Schedule-Widget-Setup.exe`
- **코드 서명이 없어 SmartScreen("Windows의 PC 보호") 경고가 뜬다.** 사용자는 "추가 정보 → 실행"을
  눌러야 한다. 없애려면 코드 서명 인증서(연 수십만 원)가 필요하므로 지금은 릴리스 노트로 안내한다.

## 11. 자동 업데이트 (v0.1.3~)

위젯의 **화면**은 웹 배포만으로 갱신되지만, **창의 성질**(트레이·권한·Win32)을 고치면 실행 파일을
새로 깔아야 한다. 업데이터가 없으면 매번 "다시 받아서 설치하세요"를 시켜야 하고, 결국 사용자는
옛 버전에 머문다. → `tauri-plugin-updater` 도입.

**동작**: 앱을 켤 때 백그라운드로 `latest.json` 을 확인 → 새 버전이 있으면 물어보고(`src/update.rs`)
설치 후 재시작. 트레이 메뉴 `업데이트 확인` 으로 직접 확인도 가능하다.
말없이 자동 설치하지 않는 이유: 설치하면 위젯이 재시작되는데, 쓰던 중에 창이 사라졌다 뜨면
고장으로 오해한다.

**서명 키 (분실 주의)**
- 개인키: `~/.tauri/schedule-widget.key`, 비밀번호: `~/.tauri/schedule-widget.pass`
- **저장소에 절대 넣지 않는다. 잃어버리면 다시는 업데이트를 밀어넣을 수 없다.** (반드시 백업)
- 공개키는 `tauri.conf.json` 의 `plugins.updater.pubkey` 에 박혀 있다.
  서명이 맞지 않는 파일은 설치되지 않는다 — 가짜 업데이트 차단.

**릴리스 절차** (PowerShell)
```powershell
# 1) tauri.conf.json + Cargo.toml 의 version 을 올린다
# 2) 서명 키를 환경변수로 주입해 빌드 → .exe 와 .exe.sig 가 나온다
$env:TAURI_SIGNING_PRIVATE_KEY = Get-Content "$env:USERPROFILE\.tauri\schedule-widget.key" -Raw
$env:TAURI_SIGNING_PRIVATE_KEY_PASSWORD = Get-Content "$env:USERPROFILE\.tauri\schedule-widget.pass" -Raw
cd desktop; npm run build

# 3) latest.json 을 만든다 (signature = .sig 파일 내용, url = 그 릴리스의 고정 파일명 주소)
# 4) 설치 파일과 latest.json 을 함께 릴리스에 올린다
gh release create widget-v0.1.4 "$env:TEMP\Schedule-Widget-Setup.exe" "$env:TEMP\latest.json" ...
```

**함정 (실제로 겪음)**
1. **`latest.json` 에 BOM 이 붙으면 업데이트가 조용히 깨진다.** PowerShell 5.1 의 `-Encoding utf8` 은
   BOM 을 붙이는데, Rust(serde_json) 는 BOM 을 만나면 파싱에 실패한다.
   → `[System.IO.File]::WriteAllText(path, json, (New-Object System.Text.UTF8Encoding($false)))`
2. **`$env:VAR = ""` 는 PowerShell 에서 변수 삭제**다. 비밀번호 없는 키를 쓰면 서명 단계에서
   "Wrong password" 로 실패한다. → 키에 비밀번호를 걸어 두는 편이 낫다.
3. `latest.json` 의 `url` 은 **그 릴리스 태그의 주소**로 박는다. `latest/download/...` 로 쓰면
   다음 릴리스가 나왔을 때 옛 매니페스트가 새 파일을 가리키게 되어 버전과 파일이 어긋난다.
